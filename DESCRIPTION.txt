1. PROBLEM STATEMENT
═══════════════════════════════════════════════════════════════════════════════

Healthcare institutions and patients face critical challenges:

• Healthcare Fragmentation: Data scattered across multiple disconnected systems
  without unified workflows. Hospitals use separate systems for appointments,
  reports, prescriptions, and patient data.

• Patient Confusion: Patients receive medical reports filled with complex   
  jargon, medical terminology, metrics, and reference ranges they don't
  understand. No plain-language explanations.

• Healthcare Provider Burden: Doctors waste valuable time on administrative
  tasks (managing appointments, reviewing reports, tracking prescriptions)
  instead of patient care.

• Manual Report Analysis: Extracting metrics from medical reports is
  error-prone, time-consuming, and not scalable. Different labs use different
  formats (PDF, images, scanned documents).

• Data Silos: No secure, unified system for hospitals, laboratories, and
  patients to share medical information efficiently.

• Poor Medication Adherence: Patients forget to take medications on time,
  leading to poor health outcomes and hospital readmissions.

• Lack of Intelligent Insights: Existing systems don't provide AI-powered
  health recommendations or contextual understanding of medical data.


═══════════════════════════════════════════════════════════════════════════════
2. SOLUTION: RAGNOSIS PLATFORM
═══════════════════════════════════════════════════════════════════════════════

RAGnosis is a unified, AI-powered medical report analysis and hospital
management platform that solves all these problems:

CORE CAPABILITIES:
• Automatically extracts text from medical reports (PDF, JPG, PNG) using OCR
• Identifies and extracts 28+ key medical metrics with high accuracy
• Generates AI-powered summaries in plain English patients can understand
• Calculates Health Score (0-100%) with explained formulas and recommendations
• Powers an intelligent RAG-based chatbot that answers patient health questions
• Integrates complete hospital workflow in a single platform
• Tracks medication reminders with push notifications to improve compliance
• Offers secure, role-based access for Patients, Doctors, and Hospital Staff
• Provides trend analysis and health tracking over time with interactive charts
    • Works seamlessly across web, mobile (iOS/Android), and desktop platforms

4. AI MODELS IN DETAIL - HOW THEY WORK
═══════════════════════════════════════════════════════════════════════════════

MODEL 1: TESSERACT OCR (Optical Character Recognition)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose:
  Convert scanned image PDFs and JPG images into machine-readable text

How It Works:
  1. Image is preprocessed (denoised, resized, straightened)
  2. Pixels are analyzed to identify character patterns
  3. Optical character recognition engine converts patterns to text
  4. Handles multiple languages and fonts
  5. Returns extracted text string

Why It's Needed:
  Medical reports come in many formats:
  - Scanned documents from hospitals/labs
  - Faxed lab reports converted to JPEG
  - Printed PDFs photographed with mobile phones
  Tesseract handles all these cases.

Success Rate:
  - High-quality scans (150+ DPI): 95%+ accuracy
  - Low-quality/handwritten: Works but lower accuracy (users can correct)


MODEL 2: BIOBERT (Biomedical BERT)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose:
  Understand medical terminology, extract meaningful information from reports

How It Works:
  1. Pre-trained on PubMed corpus (4.5M biomedical research articles)
  2. Learns medical vocabulary and entity relationships
  3. Fine-tuned for named entity recognition (NER) in medical text
  4. Identifies medical entities: drug names, diseases, metrics, values
  5. Provides context-aware embeddings for semantic understanding

Why It's Needed:
  Generic NLP models don't understand medical context:
  - "Hemoglobin = 11.2" might be labeled as "UNKNOWN" by generic models
  - BioBERT recognizes this as a medical measurement
  - Understands relationships: "elevated cholesterol" = HIGH status
  - Knows which values are abnormal

Example:
  Input: "Patient has hemoglobin of 11.2 g/dL and WBC count of 7.8 k/µL"
  BioBERT Output: {hemoglobin: 11.2, wbc: 7.8, ...}


MODEL 3: BART (Bidirectional Encoder Representations from Transformers)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose:
  Generate concise, plain-English summaries of complex medical reports

How It Works:
  1. Takes raw medical report text as input
  2. Uses denoising autoencoder architecture
  3. Pre-trained on large document summarization corpus
  4. Generates abstractive summaries (rewrites, not just extracts)
  5. Outputs patient-friendly explanation

Why It's Needed:
  Raw medical reports are dense and technical:
  - Filled with abbreviations, units, reference ranges
  - Patients don't understand medical jargon
  - BART converts: "WBC: 7.8 k/µL (4.5-11.0)" → "White blood cell count is normal"

Example:
  Input: "CBC Report - Hemoglobin: 11.2 g/dL (ref: 12.0-16.0), RBC: 3.8M/µL..."
  BART Output: "Your hemoglobin is slightly low at 11.2 g/dL. This may indicate
               mild anemia. Consider iron-rich diet and consult doctor."


MODEL 4: FAISS (Facebook AI Similarity Search)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose:
  Retrieve relevant medical knowledge instantly to power RAG chatbot

How It Works:
  1. Medical knowledge base stored as text documents (JSONL)
  2. Sentence Transformers convert each document to vector embedding (384-dim)
  3. FAISS indexes all vectors for fast similarity search
  4. When user asks question:
     a. Query converted to same vector space
     b. FAISS finds most similar documents in <1ms
     c. Documents passed to LLM as context
  5. LLM generates response based on retrieved documents + user data

Why It's Needed:
  Chatbot needs instant access to medical knowledge:
  - Can't do keyword search (user says "I feel tired", doc says "fatigue")
  - Can't scan 500 documents for every query
  - FAISS enables semantic search + instant retrieval
  - Ensures responses are grounded in medical knowledge

Example:
  User: "Why is my glucose high?"
  FAISS retrieves: [doc: diabetes explained, doc: glucose management, ...]
  LLM: "High glucose (fasting >126 mg/dL) can indicate diabetes. Your levels
        show [user's actual value]. Consult your doctor..."


MODEL 5: LLAMA 3 VIA GROQ API (Large Language Model)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Purpose:
  Generate conversational, intelligent responses about patient health

How It Works:
  1. Receives context:
     - User question
     - Retrieved medical documents (via FAISS)
     - Patient's actual medical report data
     - System prompt (instructions to stay medically accurate)
  2. LLaMA 3 8B model processes context
  3. Generates response token-by-token
  4. Groq LPU hardware enables 500+ tokens/second (vs GPU: 50 tokens/sec)
  5. Streaming response sent to user in real-time

Why It's Needed:
  Patients need intelligent, conversational AI:
  - Can ask follow-up questions naturally
  - Receives personalized responses based on their data
  - Explanations are in conversational language, not technical
  - Groq's LPU provides instant responses (not batch processing)

Example:
  User: "Should I be worried about my cholesterol level?"
  System Context: FAISS found cholesterol management docs, Patient's LDL=180
  LLaMA 3: "Your LDL cholesterol is 180 mg/dL, which is considered high (above
            100 is suboptimal). This increases cardiovascular risk. Recommendations:
            1. Reduce saturated fat, 2. Increase fiber, 3. Consult cardiologist."

5. REPORT PROCESSING PIPELINE (11-STEP WORKFLOW)
═══════════════════════════════════════════════════════════════════════════════

STEP 1: USER UPLOADS REPORT
━━━━━━━━━━━━━━━━━━━━━━━━━━
Patient uploads medical report:
  - Via web browser: Drag-drop or file selector
  - Via mobile: Camera or gallery selection
  - File types: PDF, JPG, PNG, JPEG
  - Max size: 10MB

STEP 2: FILE VALIDATION
━━━━━━━━━━━━━━━━━━━━━━
Backend validates:
  - File type (whitelist: pdf, jpg, png, jpeg)
  - File size (< 10MB)
  - Virus scan (optional)
  - Unique filename generation

STEP 3: SECURE STORAGE
━━━━━━━━━━━━━━━━━━━━━
  - File stored in /backend/uploads/ directory
  - Filename encrypted with UUID
  - Original filename saved in metadata
  - File path indexed in MongoDB

STEP 4: TEXT EXTRACTION
━━━━━━━━━━━━━━━━━━━━━━
Depending on file type:

  If PDF (native text):
    - pdfplumber extracts text page-by-page
    - Columns and tables parsed
    - Result: Plain text string

  If Image (scanned/photo):
    - Tesseract OCR processes image
    - Handles multiple languages
    - Result: Plain text string

STEP 5: LANGUAGE DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━
  - Detect language (English, Hindi, etc.)
  - Optional: Translate to English
  - Save language metadata

STEP 6: MEDICAL METRIC EXTRACTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Three-pronged strategy using BioBERT + Regex:

  Strategy 1 - Keyword + Separator:
    Regex: "Hemoglobin\s*[:=]\s*(\d+\.?\d*)"
    Found: "Hemoglobin: 11.2" → hemoglobin = 11.2

  Strategy 2 - Table Row Scan:
    Look for metric keyword on line → take first number
    "WBC Count    7800    10^3/uL"
    Found: "wbc = 7800"

  Strategy 3 - Unit Normalization:
    Raw CBC values are in cells/µL, normalize to k/µL
    7800 cells/µL → 7.8 k/µL (divide by 1000)
    220000 Platelets → 220 k/µL

  Result: 28+ extracted metrics with values and units

STEP 7: UNIT NORMALIZATION
━━━━━━━━━━━━━━━━━━━━━━━━
  - Convert all values to standard units
  - Example: mg/dL, g/dL, k/µL, mIU/L
  - Handle different lab conventions
  - Store normalized + original values

STEP 8: AI SUMMARIZATION (BART)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  - BART generates plain-English summary
  - Summary explains:
    * Which metrics are abnormal
    * What the abnormalities mean
    * Risk level (low, moderate, high)
    * Suggested follow-up actions

STEP 9: HEALTH ANALYSIS & SCORING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  - Compare each metric against clinical reference ranges
  - Determine status: LOW / NORMAL / HIGH
  - Calculate health score:
    * 0-20%: Critical (multiple abnormal values)
    * 20-40%: Poor (several abnormal values)
    * 40-60%: Fair (some abnormal values)
    * 60-80%: Good (1-2 slightly abnormal)
    * 80-100%: Excellent (all normal)

STEP 10: GENERATE RECOMMENDATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
For each metric:
  - Provide actionable tips
  - Suggest lifestyle changes (diet, exercise)
  - Recommend follow-up tests
  - Link to educational resources

STEP 11: STORE & INDEX
━━━━━━━━━━━━━━━━━━━━━
Store in MongoDB:
  - Report metadata (filename, date, type)
  - raw_text (full extracted text)
  - metrics (all 28+ values)
  - summary (BART-generated)
  - health_score (0-100%)
  - recommendations (actionable tips)

Index for Chatbot:
  - Report text added to FAISS vector index
  - User can chat about "my report"
  - RAG retrieves report context for accurate answers

9. SECURITY & AUTHENTICATION
═══════════════════════════════════════════════════════════════════════════════

JWT (JSON WEB TOKEN) FLOW
━━━━━━━━━━━━━━━━━━━━━━━
1. User logs in with username and password
2. Backend validates password against bcrypt hash in database
3. Backend generates JWT token containing:
   - user_id
   - role (Patient/Doctor/Receptionist)
   - email
   - exp (expiration: 24 hours)
   - iat (issued at timestamp)
4. Token encrypted with Flask-JWT-Extended
5. Token returned to frontend
6. Frontend stores in localStorage (web) or secure storage (mobile)
7. All subsequent API calls include JWT in Authorization header:
   Authorization: Bearer <JWT_TOKEN>
8. Backend validates JWT on every request:
   - Checks signature
   - Verifies expiration
   - Extracts user_id
9. If valid, request processed; if invalid, return 401 Unauthorized

ROLE-BASED ACCESS CONTROL (RBAC)

DATA PROTECTION
━━━━━━━━━━━━━━
Password Security:
  - Passwords hashed with bcrypt (never stored in plaintext)
  - Salt rounds: 10
  - Even database admins can't see passwords

JWT Secret:
  - Stored in .env file (not in code repository)
  - Never committed to Git
  - Different secret per environment (dev/staging/prod)

API Security:
  - All API calls use HTTPS (TLS 1.2+)
  - No HTTP endpoints
  - CORS enabled only for trusted origins
  - Rate limiting on login endpoint (prevent brute force)

Database Security:
  - MongoDB Atlas with IP whitelisting
  - Only app server IP whitelisted
  - Encryption at rest
  - Encryption in transit
  - No public internet access

File Security:
  - Uploaded files stored in /uploads with restricted access
  - Files served only to authenticated, authorized users
  - No direct URL access (files served via API with auth check)
  - Virus scanning optional

11. SYSTEM RUNTIME FLOW
═══════════════════════════════════════════════════════════════════════════════

USER UPLOADS A REPORT - COMPLETE FLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Frontend (React)
   - User selects file via UI
   - File validation (size, type)
   - Shows upload progress

2. API Call - Upload Endpoint
   - POST /api/reports/upload
   - FormData with file
   - JWT token in Authorization header

3. Backend (Flask) - Receives Upload
   - Validates JWT token
   - Checks user role and permissions
   - Validates file type/size
   - Generates UUID filename
   - Saves file to /backend/uploads/

4. Text Extraction Service
   - If PDF: pdfplumber extracts text
   - If Image: Tesseract OCR extracts text
   - Stores raw_text in temporary variable

5. Medical Metric Extraction Service
   - Runs 3 extraction strategies (regex + BioBERT + normalization)
   - Identifies report type (Blood/Diabetes/etc)
   - Extracts 28+ metrics
   - Normalizes units

6. AI Summarization (BART)
   - Passes raw_text to BART model
   - Generates plain-English summary
   - Explains abnormalities

7. Health Score Calculation
   - Compares each metric against clinical ranges
   - Assigns LOW/NORMAL/HIGH status
   - Calculates weighted health score (0-100%)
   - Generates recommendations for each metric

8. Database Storage (MongoDB)
   - Creates new report document
   - Stores metadata, raw_text, metrics, summary, health_score
   - Stores recommendations array

9. RAG Indexing (FAISS)
   - Extracts report text
   - Converts to embedding vector (384-dim)
   - Adds to FAISS index
   - User can now chat about this report

10. Response to Frontend
    - Returns report_id, health_score, metric summary
    - Frontend displays results
    - User can view full report, chat about it, or upload another

CHATBOT INTERACTION - COMPLETE FLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. User types question: "Why is my glucose high?"
2. Frontend sends: POST /api/chat/ { message: "Why is my glucose high?" }
3. Backend receives request with JWT
4. FAISS Search:
   - Converts question to embedding
   - Searches for 3-5 most similar documents
   - Retrieves relevant medical knowledge
5. User Data Lookup:
   - Fetches user's latest report
   - Extracts glucose value and other metrics
6. Context Construction:
   - Combines: retrieved docs + user metrics + system prompt
   - Prompt instructs: "Answer medically accurately based on context"
7. LLaMA 3 Generation:
   - Receives full context
   - Generates response token-by-token
   - Groq LPU processes at 500+ tokens/sec
   - Response streamed to frontend in real-time
8. Response: "Your glucose is [value], which is elevated. This can indicate
   diabetes or metabolic issues. Consider: 1) consulting endocrinologist,
   2) reducing sugar intake, 3) increasing exercise..."
